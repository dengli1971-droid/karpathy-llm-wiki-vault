# 北美AI搜索大厂产品CLI化趋势，销售和开源策略向2B变现侧重，AI观测需求和竞争优劣势。Embedding模型能力差异，收购整合进展等。Dynatrace/ClickHouse/Chronosphere/Datadog/MongoDB

**日期**: 2026-06-06 | **来源**: acecamptech.com | **公司**: Alibaba Cloud, Anthropic, ByteDance, OpenAI, Tencent | **标签**: AI训练, 大模型, TMT | **Ticker**: 0700 HK, 700 HK, 9988 HK, BABA US

## 问答
### Q1: 研发和产品的重心方面，Elastic 和 Datadog、Dynatrace、Splunk相比，更倾向于长期培育开发者生态，而不是短期激进地争取客户？

这里面有几点需要说明。过去ES的销售策略确实是“bottom-up”，先获得开发者，培育开发者生态，然后再尝试将其转化为企业级客户。以前确实是这样，所以当ES转向以解决方案为主时，在客户这边会比较挣扎。客户想要的是开箱即用的解决方案，以及行业积累的内容。例如，Datadog Spark 一开始就有很多行业解决方案，像ES和日志易竞争时，他们能提供中小银行从柜面到系统的一整套监控体系，而ES提供的只是工具，解决方案方面存在问题，因此ES一直在补足这块。Datadog 主要因为在云上，提供的是一套标准的接入方案，无论哪个云，只要是在云上，就能获得一致、标准、开箱即用的产品，这也是企业客户看重的。
客户对 ES 的印象是需要很强的工程能力才能用，这也是ES以前比较挣扎的地方。现在ES转向企业客户，也在做解决方案。以前研发和产品团队有专门做 ES 架构优化的平台团队，现在公司内部组织架构有 observability、security、UI、AI 等团队，都是根据客户痛点划分研发团队，面向企业级客户。产品团队也需要做产品白皮书和面向客户的 product marketing 材料，从客户案例出发，说明功能如何解决客户问题。这是企业方向。
但在近期 AI 重心的背景下，其实并不是明显转向企业，因为所有产品现在有一个趋势，是面向 AI 开发的产品，既不是面向企业，也不是面向开发者生态，而是面向 AI 和 agent。面向 agent 有一个非常重要的趋势，就是所有产品功能都必须 API 化、CLI化，文档要写清楚每个参数怎么用，让 agent 能学习到，并放在互联网上，进入大模型的 pretrain 材料里，让大模型天生知道如何调用你的平台、组装 API，完成企业级需求。你要有足够丰富的 API，覆盖足够多场景。现在企业客户其实不需要软件定制化，不需要企业为他定制产品，所有定制化都由 AI 完成，不需要 UI 上有很多功能，agent 接入后可以直接定制化想要的内容。现在比较火的是 MCP apps，可以作为 plugin 或 extension 嵌入 cloud，用户以对话方式实现功能，比如画图表、数据分析，直接在 agent 上呈现，完成定制化功能，不需要找企业开发，直接和 agent 说就能实现。如果找企业开发，可能要等好几个迭代才能上线。所以现在的重点一定是面向 agent 做开发。

### Q2: 从 bottom-up转向有针对性地销售的时点效果？

是从 Ashutosh成为ES的 CEO 开始，他强调以销售和客户为主，之前是以产品为主。从那时候开始就变成这样，ES也做了很多销售培训。现在的效果比之前明显很多。以前ES去客户那边，销售主要做的是Elasticsearch本身的调优，除了调优之外没有其他内容可以讲。现在所有材料都聚焦于上面的功能，不再讲Elasticsearch的调优和设计，这些内容一般都不再提，就是要改变客户的传统认知，让他们理解ES是一个分布式操作系统，ES是来帮客户维护这套分布式搜索引擎。

### Q3: 现在的ES产品和方案都已经CLI化了吗？

关于CLI化ES起步比较早，产品文档早就拆分开了，所有产品描述都在doc里。两年前ES就把所有API文档单独拆出来，作为一份API文档，非常明显，这份文档是给AI看的，爬虫爬完后进入训练数据。API文档不仅变成pretrain数据，也包括很多产品功能，比如Kibana。以前Kibana是UI界面，需要用户拖拽和点击才能完成操作，现在所有操作背后都有API，点击操作都可以转化为API调用。以前这些API只在服务器内部，不开放，现在全部开放，包括云上的基础设施使用，支持terraform API，可以通过terraform完成集群部署。从部署开始，到ES里的所有功能，再到Kibana上的功能，甚至现在Kibana上有一个项目叫Dashboard API、SAP API，构造仪表板的所有功能都可以通过API实现，调API传参数就能生成可视化组件，再把这些组件组成dashboard。现在有build dashboard这种mcp app，可以让数据分析功能在cloud产品上以对话方式实现和构建。

### Q4: Elasticsearch和Kibana的活跃开发者数量和付费转化率，以及转化率提升的趋势如何？

ES的活跃开发者其实很少。开源已经变成获取用户的手段，没有哪个公司真正做开源，也没有哪款产品依靠活跃开发者提供支持，全部都是商业公司掌控90%以上的代码提交。所谓的开源活跃提交者，现在主要是腾讯和阿里，腾讯、阿里贡献比较多，可能还有其他第三方公司。现在很少接受活跃开发者的提交，也没有开发者能像以前那样看完整个开源产品代码并提供显著贡献。如果有贡献，通常是因为产品上云，是云厂商员工提的额外开发。例如open search上，字节跳动提交比较多，是因为他们在open search上提供服务。像Doris、SelectDB这些，都是商业公司把持代码，很少有活跃开发者。开源现在只是一个幌子，产品是开源的，主要是license开源，可以让用户免费用，际上转化率非常低。国内开源公司，尤其互联网公司，无论用ES还是Doris，肯定是免费用，付费用户转化水平很低。能转化过来的通常是因为合规需求，比如FSI客户需要售后合规。
现在开源公司也慢慢明白了，一定要有开源版本和闭源版本，闭源版本才能转化收费，开源版本基本不能转化。只有付费的闭源版本才能收费。比如Miro，用户很少用Miro，只有Ziliz才能收费，Ziliz和Miro本身功能差别很大。MongoDB和ClickHouse也是一样，开源版本大家都用，云上的版本才有很多开源版没有的功能。ClickHouse云上的ClickStack和MongoDB云上的Atlas都有很多开源版没有的功能。现在的玩法都是靠云来转化。

### Q5: 怎么判断Datadog在观测领域对ES的份额的影响？

Datadog有一个限制，它只是云上的可观测解决方案，对云下没有提供产品。现在有一个趋势，随着AI浪潮，很多服务需要搭在云上，包括token消耗、租赁CPU做计算，很多原先客户有新的项目会迁到云上，云的规模会扩大。云规模扩大一定会让Datadog云上的观测营收增加。但对于传统无法上云的客户，Datadog影响不到。

### Q6: 是否了解Chronosphere的情况？怎么看它和ClickHouse的优劣势和前景？

它有两个优势：第一是降本增效能力，包括对数据点做采样，减少数据量。很多遥测数据需要采样，比如秒级、毫秒级采集的数据要上卷到分钟级、五分钟级，去看更粗的趋势，历史数据会丢弃，这里需要采样算法帮助有效上卷，减轻存储压力。Chronosphere在数据进入存储前就能做过滤、聚合、塑形、采样，这些能力很强。第二，它原生做GPU解决方案，非常了解不同GPU型号及相关外设，包括铜缆、连接、交换机等，整套指标体系都做了，知道哪些指标重要，代表什么意义，并制定了对应规则，做了GPU级的解决方案，所以能得到Anthropic、OpenAI等公司的认可。ClickHouse只是一个OLAP数据库，强调数据分析能力。

### Q7: 未来一年云上的观测需求增长一定会快过OP这边？

这个不好说。云上的增速不太好判断，现在很多OP的业务，企业都在做agent，这会涉及到新的observability，比如LLM的observability。OP这块也会有增量。ES在指标方面刚出的PromQL支持，而且还没有看到有大型参考客户可以用来讲故事。ES的IUM这块肯定不如Datadog强。Anthropic、OpenAI这些模型大厂本身是2C场景，特别在意IUM。

### Q8: ES在IUM这边有没有发力？

ES很早之前就有这个产品，能力也是对标Datadog做的，但收入一直不高。

### Q9: 云这边和OP去年（2025年）都涨价了，但营收没有体现出来，怎么理解这个原因？

在中国市场和客户谈价时，客户根本不认可涨价。第一年销售只是和客户讨论价格，但最终合同基本还是按去年（2025年）的价格签，通常会说第二年再谈涨价。第一年没有客户愿意涨价，销售也无法强推，没有强议价能力。第一年只是通知客户涨价，但合同基本保留去年（2025年）的价格。今年（2026年）又涨价了，但还是不够用。有不少客户在涨价前签了multi-year的合同，通常是三年期，这些合同在涨价前签订，因此未来两三年内，这部分-year合同的价格不会上涨。需要等到去年开始算起的两三年后，才能逐步看到价格变化的影响。云服务也涨价了，而且云的涨价速度比OP要快一些。

### Q10: 云厂商最近也在涨价，如果ES无法将成本传导给客户只能自己吸收成本增加？

ES没有成本增加。国内ES并没有采购云厂商的云服务，只是授权商标，所有基础设施都是他们自己建设的。在国外，ES才会采购并建设Elastic Cloud。国内没有Elastic Cloud，只有阿里云和腾讯的ES Service，那是他们自己的服务。

### Q11: ES去年对Jina的收购整合情况如何？有没有显著提升产品或对客户的吸引力？

Jina补足了ES没有Embedding模型的短板。最近Jina发布了几个模型，包括Omni，这是一个多模态模型，可以把视频、音频等放在同一个向量空间，用文本向量搜索视频和音频。这个产品刚推出，公司还没有明确如何推广新纳AI的销售，目前是把它放在云上作为推理服务销售，按照token方式收费，Jina以前也是这样运作的。Jina的Embedding模型收入在其财务中并不高。模型收费有两个问题：一是如何收费，因为不像软件可以抽取核心功能收费，模型本身就是一组参数，很难拆分出商业版，开源版本已经满足大部分企业用户需求。商业版主要提供支持服务，比如fine tuning、auto scaling、扩展等最佳实践，都是服务型功能，没有办法绑定用户。对于线下客户，唯一能收费的方式是把模型部署在云上，提供自动扩缩容和低延迟推理能力，用户只会因为这点购买推理服务，还是按照token方式收费。这种收费方式很难体现在产品销售环节，也难以促进ES的销售，拥有这个模型并不能让客户购买更多节点，只是满足部分客户的推理需求。ES还没有看到非常明确的点能帮助整体产品销售。
Jina和MongoDB收购的情况不太一样。Jina主打开源和小模型，商业客户愿意为模型付费的多是法律、金融、科研等领域，但Jina没有针对这些领域做垂直优化，也没有做几十亿参数的大模型，主要强调性价比和低资源部署的通用Embedding模型，没有像MongoDB收购的Voyage AI那样面向企业用户。目前来看，Jina只是补足了产品线能力，但对企业客户吸引力和整体产品销售增长没有明显提升。
