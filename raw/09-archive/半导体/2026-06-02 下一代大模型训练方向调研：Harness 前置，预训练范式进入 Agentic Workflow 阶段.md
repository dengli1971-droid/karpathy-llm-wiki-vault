# 下一代大模型训练方向调研：Harness 前置，预训练范式进入 Agentic Workflow 阶段

**日期**: 2026-06-02 | **来源**: acecamptech.com | **公司**:  | **标签**: AI推理, AI训练, 大模型, 数据中心, HBM, TMT

## 核心观点
- 新一代 frontier model 的提升不只是参数和 token 规模扩张，更关键是 Harness 能力从后训练阶段前置到预训练阶段。通过引入 trajectory data，模型从训练初期就学习复杂任务的规划、拆解、工具调用、记忆管理、验证和回滚过程，从而更原生地具备 Agentic workflow 能力。
- 训练侧，算力需求继续上行，但主要来自模型规模和数据量增长，而非底层架构突变；Transformer 短期仍是主流。随着模型规模扩大，跨数据中心互联、数据清洗和训练调试能力成为核心壁垒。
- 推理侧，Agentic workflow 会推高 token 消耗和 KV cache 需求。短期 DRAM 作为 HBM 缓存更现实，长期若历史对话和任务状态持续累积，SSD 也可能承接更多 KV cache 或 workflow 数据，带来存储需求弹性。

## 正文

近期笔者针对新一代大模型的训练方向进行了一轮调研。整体感受是，市场在讨论下一代模型时，仍然容易把重点放在“参数会不会更大、训练 token 会不会更多、算力消耗会不会继续上升”这些线性外推问题上。但从产业一线反馈来看，下一代模型真正值得关注的变化，可能并不只是规模继续扩张，而是 Harness 能力正在从后训练阶段前置到预训练阶段。

这里的 Harness，可以理解为模型围绕复杂任务进行规划、拆解、工具调用、记忆管理、验证、回滚和多步骤执行的综合能力。过去一代模型更多是先通过大规模预训练获得通用知识和基础能力，再通过后训练阶段补充 Harness 能力；而下一代模型的重要方向，是让模型在预训练阶段就直接学习复杂任务的执行轨迹。

这背后的关键数据形态是 trajectory data，即轨迹数据。它不是简单的文本、代码、图片、视频，也不只是最终答案，而是一个复杂任务从提出、拆解、执行、调用工具、验证结果到修正路径的完整过程。换句话说，模型不只是学习“答案是什么”，而是学习“一个任务如何被完成”。






一、主线判断：下一代训练的核心是 Harness 前置，而不是单纯模型变大

笔者理解，下一代大模型训练的核心变化，可以概括为一句话：

Harness 能力从后训练补充，变成预训练内化。

当前一代模型的复杂任务能力，很多是在后训练阶段形成的。模型先完成大规模预训练，获得通用知识、语言能力和基础推理能力，再通过持续预训练、指令微调、工具调用增强等方式，让模型更适合处理复杂指令、多步推理、memory recall、工具使用和 Agentic workflow。

这种方式当然有效，但它有天然边界。因为后训练是在一个已经成型的模型上继续调整，为了避免模型已有能力出现非预期退化，训练过程通常不能对底层权重做过大扰动。因此，后训练更像是在已有能力上的校准、激活和重组，而不是从底层重新塑造模型的任务执行方式。

下一代模型的变化在于，Harness 相关的 trajectory data 被直接纳入预训练。模型从训练初期就开始学习复杂任务如何被推进，而不是等模型训练完成后，再通过后训练去补“怎么做事”的能力。

这也是为什么笔者认为，下一代模型能力跃迁的关键，不只是模型尺寸继续扩大，而是 Harness 能力的形成位置发生了变化。






二、什么是 Harness：它决定模型能不能真正“做事”

如果说传统大模型的核心能力是理解、生成和推理，那么 Harness 更接近“任务执行系统”。

它包括几个层面：

第一，规划能力。模型需要把一个开放式目标拆成可执行步骤，而不是只给出静态答案。

第二，工具调用能力。模型需要知道什么时候调用代码环境、搜索、数据库、文件系统、企业软件或其他外部工具。

第三，记忆管理能力。模型需要判断哪些上下文应该保留，哪些应该压缩，哪些应该召回，哪些可以丢弃。

第四，验证和回滚能力。模型不能只生成结果，还要检查结果是否正确，发现错误后能调整路径。

第五，多步骤执行能力。模型需要在长周期任务中持续推进，而不是完成一次回答后结束。

这些能力合在一起，才构成真正的 Agentic workflow。当前很多模型看起来已经会调用工具、会拆任务，但在复杂开放任务中仍然不稳定，原因就在于 Harness 能力还不够原生，很多能力是在后训练阶段“补”出来的，而不是在预训练阶段“长”出来的。






三、为什么 trajectory data 重要：它是训练 Harness 的核心材料

Harness 能力要前置到预训练，关键就需要 trajectory data。

过去的大模型训练，核心数据形态大多是结果型语料。网页、书籍、论文、代码、问答、多模态内容，本质上都在告诉模型某个结果应该如何表达、如何回答、如何生成。

但复杂任务真正有价值的部分，往往不是最终结果，而是中间过程。

以 coding agent 为例，如果模型只学习最终代码，它学到的是“答案长什么样”；但如果模型学习完整任务轨迹，它学到的是“一个工程任务如何被推进”。这中间包括理解代码库、搭建环境、拆解需求、运行测试、定位 bug、修改代码、再次验证，以及失败后的调整路径。

这类 trajectory data 本质上就是专家工作流的数据化。它把专家如何思考、如何试错、如何调用工具、如何验证输出，变成模型可以学习的训练样本。

所以，trajectory data 不是这篇报告里的一个小概念，而是 Harness 前置的基础设施。没有高质量轨迹数据，模型就很难在预训练阶段真正学会复杂任务执行。






四、下一代模型能力跃迁：从“回答问题”走向“推进任务”

Harness 前置之后，模型能力的评价方式也会变化。

上一代模型的能力更多体现在回答质量上：能不能回答得准确，能不能表达得清楚，能不能完成常规推理，能不能理解多模态内容。

下一代模型更重要的验证场景，会是复杂 Agentic workflow。也就是说，模型能不能持续推进一个长链路任务。

当前 AI Agent 在简单、固定、确定性强的任务上已经相对成熟，例如会议安排、提醒、信息发送、价格监控、内容审核、数据标注、简单网站搭建等。这些任务结构清晰，目标明确，模型只要有合适的产品设计和足够算力，就能比较好地完成。

但复杂任务完全不同。比如分析一个强化学习训练中 loss 为什么不下降，模型需要持续交互、提出假设、设计实验、调用工具、检查代码、判断结果，并在失败后调整路径。当前模型在这类任务中仍然容易出现路径不清、验证不足、循环试错但无法收敛的问题。

如果下一代模型在预训练阶段就学习大量 Harness trajectory，它的提升不一定表现为单次回答更漂亮，而是表现为更能推进任务。它会更像一个能持续工作的执行者，而不是一个只会回答问题的助手。






五、Scaling Law 仍在，但边际变量从 token 数量转向 token 结构

从调研情况看，下一代 frontier model 的参数规模和训练 token 量大概率仍会继续上行。每一代模型使用的预训练 token 量相比上一代明显增加，仍然是正常趋势。

但笔者认为，下一阶段不能简单理解为 scaling law 的机械延续。模型变大仍然重要，因为更大的模型有更高容量，可以承载更多知识、技能和任务模式。但更大的模型到底能不能转化成更强的复杂任务能力，取决于它学到的是什么数据。

如果训练数据仍然主要是静态文本和结果型语料，模型能力提升可能更多体现在知识覆盖、语言表达和常规推理上。但如果训练数据中包含大量高质量 Harness trajectory，模型才更可能在复杂开放任务上出现阶跃式提升。

所以，下一代模型训练的关键可能不是“更多 token”这么简单，而是“更有结构的 token”。未来模型公司之间的差距，可能会从训练 token 总量，进一步扩展到 trajectory data 的质量、覆盖领域、可验证性和清洗能力。






六、训练工程：Harness 前置会提高数据清洗和训练组织难度

Harness 前置听起来是训练范式变化，但落到工程上，它会显著提高训练难度。

首先，trajectory data 比普通文本更难清洗。普通文本的质量问题相对容易识别，但复杂任务轨迹里，错误可能隐藏在中间步骤中。例如，某个工具调用路径看似合理，但其实是无效试错；某个验证过程看似完整，但实际没有证明结果正确。如果这些低质量轨迹进入预训练，模型可能学到错误的任务执行习惯。

其次，多模态和工具调用轨迹会让数据结构更复杂。模型不只是学习文本，还要学习代码、文件、环境状态、工具返回结果、上下文压缩和任务链接关系。这些都增加了数据整理和训练调度难度。

第三，frontier model 训练本身越来越依赖跨数据中心组织能力。单一数据中心越来越难完整承载下一代模型训练所需的加速卡规模、数据吞吐和存储调度。虽然行业中也在探索减少跨数据中心通信的方法，例如并非每个 batch 都进行梯度同步，但这些方法目前还不够成熟。现阶段，大规模预训练仍然高度依赖跨数据中心连接。

因此，模型公司的竞争不只是“谁有更多 GPU/TPU”，而是“谁能把更多加速卡、数据管线、存储系统和训练任务稳定组织起来”。






七、训练周期：模型不一定完全收敛，调试能力成为关键变量

从调研反馈看，下一代模型训练通常仍受到明确发布时间窗口约束。公司很难让一个模型训练无限期延长，通常需要在几个月内完成主要训练。因此，模型最终往往不是等到完全收敛才发布，而是在效果、成本和时间之间做平衡。

训练失败本身现在已经相对少见，但非预期结果仍会显著增加调试成本。比如 loss 异常、多模态数据质量问题、数据污染、某些能力方向不达预期，都可能让原本预留的调试时间被明显拉长。

当 Harness trajectory 进入预训练后，调试难度可能进一步上升。因为模型表现不佳时，问题未必来自算力或模型结构，也可能来自轨迹数据质量、任务分布、工具调用路径、验证样本设计，或者数据中隐含的错误工作流。

所以，下一代模型公司的训练能力不能只看硬件规模。真正关键的是端到端工程能力：能不能稳定训练，能不能快速定位异常，能不能调整数据配比，能不能回滚有问题的数据，能不能在发布时间窗口内完成模型交付。






八、对应用侧的含义：Agentic workflow 会成为 Harness 的核心落地场景

Harness 前置的最终目的，是让模型更好地完成 Agentic workflow。

未来企业内部的工作流，可能会从“人直接完成任务”逐步转向“人管理多个 Agentic workflow”。一个员工不一定亲自完成所有执行动作，而是同时管理多个 agent，由模型承担更多执行、跟踪、验证和交付环节。

这里最重要的变化不是自动化更多简单任务，而是复杂任务的边界被打开。简单任务今天已经有不少可以落地，真正的增量来自模型能否处理更开放、更长周期、更需要判断力的任务。

如果下一代模型的 Harness 能力显著提升，那么 coding、科研辅助、金融分析、法律文书、企业运营、数据分析等场景，都会从“AI 辅助生成”走向“AI 持续执行”。

这会显著改变企业对模型的使用方式：模型不再只是一个问答入口，而是一个可以被分配任务、持续跟踪任务、最终交付结果的执行层。






九、对基础设施的含义：Harness 越强，推理侧资源消耗越大

Harness 能力提升后，基础设施需求会被进一步放大。

训练侧比较直接。模型变大、数据变多、trajectory data 和多模态数据比例提升，会继续拉动 GPU/TPU、跨数据中心互联、高性能存储、数据管线和电力需求。

但更大的弹性可能在推理侧。因为 Harness 越强，用户越会让模型执行长周期、多步骤、多 agent 的复杂任务。这会带来三类资源压力。

第一，token 消耗增加。复杂任务需要更多 planning、memory recall、工具调用、验证和修正过程。

第二，KV cache 压力增加。Agentic workflow 会带来更高并发和更多上下文管理需求。即使单个 sub-agent 的上下文窗口可能变小，总任务层面的 KV cache 和内存管理压力仍会上升。

第三，存储层级更重要。短期看，DRAM 作为 HBM 的缓存层是更现实的方案，通过把大部分 KV cache 从 HBM offload 到 DRAM，可以提升并发度并降低加速卡资源成本。长期看，如果历史对话、任务状态和长周期 workflow 持续累积，SSD 也可能承担更多 KV cache 或任务状态数据。

所以，Harness 前置不是一个单纯的模型训练问题。它会让模型更会做事，而模型更会做事以后，用户就会让模型做更多事。最终被放大的，是推理侧 token、KV cache、DRAM、SSD 和内存池需求。






结论

笔者对下一代大模型训练方向的理解，可以概括为三句话。

第一，下一代模型训练的核心变化是 Harness 前置。

复杂任务执行能力不再主要依赖后训练阶段补充，而是开始进入预训练阶段内化。

第二，trajectory data 是 Harness 前置的关键训练材料。

谁能获得更多真实、复杂、可验证的专家工作流数据，谁就更可能在复杂开放任务上形成能力差异。

第三，Harness 能力提升会放大 AI 基础设施需求。

训练侧受益于模型和数据规模扩张，推理侧则受益于复杂 Agentic workflow 带来的 token、KV cache、DRAM 和长期存储需求增长。

因此，下一代大模型训练的关键词不是单纯的“更大”，而是：

Harness 前置、轨迹数据预训练、复杂任务工作流化。
